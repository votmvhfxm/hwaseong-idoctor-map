# 화성 아이닥터 맵 — 데이터 계약 문서 (v1)

> **참고용 문서 — 실제 구현과 다른 점 있음.** 이 문서는 팀원(윤정빈)이 인수인계용으로 작성한 초기 UI 셸(`화성_아이닥터맵.html`)의 데이터 계약이며, 그 UI 디자인 요소는 현재 프로젝트(index.html/css/js)에 병합됐다. 다만 실제 지도·데이터 아키텍처는 이 문서의 `fetchData()`/`Zone.boundary` 단일 모델보다 더 진행되어 있다:
> - zones/clinics는 `js/data.js`에 정적 표본으로 존재(이 문서가 말하는 실시간 `fetchData()` 비동기 파이프라인은 미구현 — 로딩/에러/빈 상태 목업도 적용 안 함, 실제로 비동기가 아니라서 정직성 원칙상 뺐음).
> - 행정경계는 이 문서의 `Zone.boundary`(단일 GeoJSON) 대신, 화성시 28개 읍면동 실경계(`js/geo.js`)를 10개 zone으로 근사 그룹핑해 표시(자세한 내용은 CLAUDE.md 참고).
> - 지도는 `api/config.js`(Vercel 환경변수)로 카카오맵 JS 키를 자동 연결하고, 실패 시 수동 키 입력 UI로 폴백.
> - 이동시간은 `api/directions.js`(Vercel 서버리스 프록시)로 카카오모빌리티 실연동, 미배포 시 데모 추정치로 폴백.
>
> 아래 원문은 원래 의도(스키마·역할 분담·공공데이터 매핑 가이드)를 이해하는 참고 자료로만 남겨둔다.

> UI(윤정빈)와 데이터/API(팀장) 사이의 **인터페이스 계약서**입니다.
> UI는 오직 이 문서의 스키마에만 의존합니다. 팀장은 아래 형태로 데이터를 돌려주는 **함수 하나(`fetchData`)만** 구현하면 연동이 끝납니다.
> UI 코드에서 교체 대상은 HTML 파일의 `// [데이터 레이어]` 구간뿐입니다.

## 0. 인수인계 안내 (이어서 작업하는 법)

**함께 오는 파일 2개**
1. `화성_아이닥터맵.html` — 표본 데이터로 작동하는 UI 셸 (이 문서의 스키마에만 의존)
2. `화성_아이닥터맵_데이터계약서.md` — 이 문서

**역할 분담**
- **UI 담당(윤정빈):** 화면·리스트·필터·상태·상세·커버리지 (이미 구현됨, 계속 다듬음)
- **API/지도 담당(팀장):** 아래 두 곳만 채우면 됨
  - (A) `fetchData()` → 프록시로 공공데이터 호출 후 이 문서의 스키마로 반환
  - (B) `buildMap()` → 개념도(SVG)를 **카카오맵**으로 교체 (지도: 카카오 확정)

**Claude로 이어서 작업하려면:** 위 HTML 파일과 이 계약서를 Claude 대화에 함께 붙여넣고, "이 계약서 스키마에 맞춰 카카오맵 연동과 fetchData 프록시를 이어서 작업해줘"라고 요청하면 됩니다. 두 파일이 모든 맥락을 담고 있습니다.

## 1. 핵심 계약 — `fetchData()`

UI는 시작 시 아래 함수를 호출하고, 성공/실패에 따라 화면 상태를 바꿉니다.

```
async function fetchData(): Promise<{
  zones:   Zone[];      // 행정동 목록
  clinics: Clinic[];    // 진료기관 목록
  updatedAt: string;    // 데이터 기준 시각 (ISO8601, 예: "2026-07-03T14:05:00+09:00")
}>
```

- **성공**: 위 객체를 `resolve` → UI가 지도·리스트·커버리지를 렌더.
- **결과 없음**: `clinics: []` 로 정상 반환 → UI가 "조건에 맞는 곳 없음(빈 상태)"을 표시.
- **실패**: `reject(에러)` 또는 throw → UI가 "정보를 불러오지 못했어요 + 다시 시도"(오류 상태)를 표시.

> UI는 로딩 스켈레톤 → 위 결과에 따라 상태 전환까지 이미 구현돼 있습니다. 팀장은 반환값만 맞추면 됩니다.

## 2. `Clinic` 스키마 (진료기관)

| 필드 | 타입 | 필수 | 설명 | 공공데이터 출처(예) |
|---|---|:--:|---|---|
| `id` | string | ✅ | 고유 식별자 | 심평원 암호화 요양기호 |
| `name` | string | ✅ | 기관명 | 심평원 병원정보(`yadmNm`) |
| `type` | `"소아과"` \| `"달빛"` | ✅ | 진료기관 유형 | 표시과목/기관종별로 매핑 |
| `zoneId` | string | ✅ | 소속 행정동 id (Zone.id와 일치) | 주소·읍면동으로 매핑 |
| `lat` | number | ✅ | 위도 | 심평원 `YPos` |
| `lng` | number | ✅ | 경도 | 심평원 `XPos` |
| `phone` | string | ▲ | 전화번호 | 심평원/상세정보 |
| `address` | string | ▲ | 주소 | 심평원 병원정보 |
| `hours` | `{open:number, close:number}` 또는 요일별 | ✅ | 진료시간(24h 정수). 표본은 단일 시간, 실서비스는 요일별 권장 | 심평원 의료기관별상세(진료시간) |
| `todayOpen` | boolean | ▲ | 오늘 실제 진료 여부(실시간). 있으면 UI가 우선 사용 | 응급의료 E-Gen 실시간 |
| `intake` | `"현장"` \| `"전화"` \| `"앱"` | ▲ | 접수 방식 | 기관 정보/자체 |
| `ageFrom` | string | ▽ | 진료 가능 연령 표기(예: "0개월~") | 자체/상세 |
| `wait` | number \| null | ▽ | 예상 대기(분). 없으면 `null` | (제휴 시) 실시간 대기 |

- 필수(✅): 없으면 그 기관은 표시 불가.
- 권장(▲): 없어도 동작하나 상세·안내 품질 저하.
- 선택(▽): 있으면 태그/상세에 노출, 없으면 자동 생략.

### 요일별 진료시간(실서비스 권장 형태)
```
hours: {
  mon:[9,18], tue:[9,18], wed:[9,18], thu:[9,18], fri:[9,18],
  sat:[9,13], sun:null, hol:null   // null = 휴진
}
```
> 표본 단계에서는 `{open,close}` 단일 형태를 씁니다. 요일별로 확장하면 UI 렌더 함수 한 곳만 그에 맞춰 조정하면 됩니다(협의).

## 3. `Zone` 스키마 (행정동)

| 필드 | 타입 | 필수 | 설명 | 출처(예) |
|---|---|:--:|---|---|
| `id` | string | ✅ | 행정동 고유 id | 행정동 코드 |
| `name` | string | ✅ | 행정동 이름 | 통계/화성시 |
| `lat` | number | ✅ | 중심 위도 | 경계 중심점 |
| `lng` | number | ✅ | 중심 경도 | 경계 중심점 |
| `pop` | number | ✅ | 영유아 인구(커버리지 계산용) | 화성시/통계청 |
| `boundary` | GeoJSON | ▽ | 경계 폴리곤(히트맵을 실제 동 모양으로) | 공공데이터/SGIS |

> `boundary`가 있으면 지도에 행정동 경계 히트맵을 그릴 수 있습니다(선택). 없으면 중심점 기준으로 표현.

## 4. UI가 계산하는 파생값 (API가 줄 필요 없음)

아래는 UI가 스키마로부터 직접 계산합니다. **API는 원자료만** 주면 됩니다.

- `openNow` — 현재 시각이 `hours` 범위 안인지(또는 `todayOpen` 우선).
- `closingSoon` — 마감 1시간 이내인지 → "마감 임박" 태그.
- `distance` — 사용자가 고른 기준 위치와 `lat/lng`로 계산.
- `coveredZone` / 커버리지 % — 각 동에 열린 기관이 있는지로 계산.
- 공백 랭킹 / 배치 시뮬 결과.

## 5. 공공 API → 필드 매핑 가이드 (팀장용)

1. **심평원 병원정보서비스** → `id, name, type, address, lat(YPos), lng(XPos)`
   (경기도 시도코드 + 화성시 시군구코드 + 소아청소년과 진료과목코드로 필터)
2. **심평원 의료기관별상세정보서비스** → `hours(진료시간), phone`
   (병원정보에서 받은 요양기호로 조회)
3. **응급의료정보 E-Gen** → `todayOpen`(달빛/응급 실시간 운영), 야간·휴일 진료 보완
4. **화성시/통계청** → `Zone.pop`(영유아 인구), `boundary`(경계)

> 진료과목·시도/시군구 코드는 심평원 보건의료빅데이터개방시스템 코드조회에서 확인.

## 6. 상태·에러 규약 (UI 동작)

| 상황 | 팀장이 반환 | UI 표시 |
|---|---|---|
| 정상 | `{zones, clinics, updatedAt}` | 지도·리스트·커버리지 |
| 결과 없음 | `clinics: []` | 빈 상태(필터 조정 안내) |
| 실패 | reject/throw | 오류 + "다시 시도" 버튼 |
| 갱신 중 | (호출 진행) | 로딩 스켈레톤 |

- `updatedAt`은 화면에 "업데이트 HH:MM · 실시간"으로 노출됨. **실시간이 아니면**(스냅샷) 그 사실을 값/문구로 구분 표기 권장.

## 7. 지도 규약 — 카카오맵으로 확정

- 좌표는 **위도(`lat`)·경도(`lng`)** 로 통일.
- UI의 지도 렌더는 `buildMap()` 한 곳에 격리 → 이 함수만 카카오맵으로 교체.
- 표본 개념도에서는 `x,y`(스키마 좌표)를 쓰며, 실연동 시 `lat,lng`로 대체.

### 카카오맵 셋업 절차 (팀장)
1. developers.kakao.com 로그인 → **애플리케이션 추가** → **JavaScript 키** 발급.
2. 앱 설정 → **플랫폼 > Web**에 **배포 도메인 등록**(등록된 주소에서만 키 동작 = 보안).
3. HTML `<head>`에 SDK 로드:
   `<script src="//dapi.kakao.com/v2/maps/sdk.js?appkey=발급키&autoload=false&libraries=services"></script>`
4. `buildMap()`에서:
   - `new kakao.maps.Map(...)`으로 지도 생성
   - `S.clinics` 순회 → `new kakao.maps.Marker({position:new kakao.maps.LatLng(c.lat,c.lng)})` 배치, 클릭 시 `openDetail(c.id)` 연결
   - `Zone.boundary`(GeoJSON)가 있으면 `kakao.maps.Polygon`으로 행정동 공백 히트맵
5. `paintMap()`/`hl()`도 카카오 오브젝트 기준으로 색·투명도 갱신하도록 대응.

> 하루 30만 건 무료(프로토타입엔 충분). JS 키는 도메인 등록으로 보호되며, 공공데이터 서비스키(§8)와는 별개다.

## 8. 보안 주의 (필수)

- 공공데이터 **서비스키는 프록시(서버) 쪽에만** 둘 것. 클라이언트 코드/깃 저장소에 절대 노출 금지(`.env` + `.gitignore`).
- 브라우저 직접 호출은 CORS로 막히므로, 프록시(서버리스 함수)를 통해 호출.
- 지도 SDK 키는 **사용 도메인 등록**으로 보호(등록된 배포 URL에서만 동작).
